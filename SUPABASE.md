# Supabase Setup Guide

This document details the steps an AI agent should take to set up Supabase for the WP Slow Plugin Scanner telemetry system.

## Overview

The telemetry system collects anonymous plugin performance data from WordPress sites to build a shared plugin compatibility database. Data is sent from the WordPress plugin to Supabase via REST API.

## Prerequisites

- Supabase account at https://supabase.com
- Access to create a new project
- Access to Supabase SQL editor

## Setup Steps

### Step 1: Create Supabase Project

1. Navigate to https://supabase.com and sign in
2. Click "New Project"
3. Fill in project details:
   - **Name**: `wp-plugin-telemetry` (or preferred name)
   - **Database Password**: Generate a strong password and save it securely
   - **Region**: Choose a region closest to your target users
4. Wait for project to provision (1-2 minutes)

### Step 2: Get API Credentials

1. In the Supabase dashboard, go to **Project Settings** (gear icon)
2. Navigate to **API**
3. Find the following credentials:
   - **Project URL**: `https://xxxxx.supabase.co`
   - **anon public key**: `eyJxxx...` (public API key)
4. Copy these credentials - they will be needed for the WordPress plugin `.env` file

### Step 3: Create Telemetry Table

Run the following SQL in the Supabase SQL editor:

```sql
-- Create telemetry table
CREATE TABLE telemetry (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    
    -- Plugin identification
    plugins TEXT[] NOT NULL,           -- All installed plugins (including plugin_tested)
    plugin_tested TEXT NOT NULL,       -- Single plugin being tested
    
    -- Speed metrics
    plugin_speed_delta REAL,           -- Additional seconds caused by plugin (NULL if error)
    baseline_site_load_speed REAL,     -- Total site load time with ALL plugins enabled
    
    -- Error tracking
    plugin_error TEXT,                 -- Error message if plugin broke site
    error_category TEXT,              -- Category: timeout|break_site|output_change|none
    
    -- Environment
    env JSONB,                         -- {"php_version": "8.2", "wp_version": "6.4"}
    
    -- Site identification
    origin TEXT,                       -- Site UUID (pia_site_uuid from wp_options)
    
    -- Timestamps
    timestamp BIGINT NOT NULL,         -- Unix timestamp
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Create indexes for common queries
CREATE INDEX idx_telemetry_timestamp ON telemetry(timestamp DESC);
CREATE INDEX idx_telemetry_plugins ON telemetry USING GIN(plugins);
CREATE INDEX idx_telemetry_plugin_tested ON telemetry(plugin_tested);
CREATE INDEX idx_telemetry_origin ON telemetry(origin);
CREATE INDEX idx_telemetry_error_category ON telemetry(error_category);
```

### Step 4: Enable Row Level Security (RLS)

1. In the Supabase dashboard, go to **Authentication** → **Policies**
2. Find the `telemetry` table
3. Enable RLS (if not already enabled)
4. Create a new policy to allow anonymous inserts:

```sql
-- Allow anonymous (and authenticated) users to insert data
CREATE POLICY "Allow anonymous inserts" ON telemetry
FOR INSERT TO anon, authenticated
WITH CHECK (true);
```

**Security Note**: This policy only allows INSERT. The anon key cannot SELECT, UPDATE, or DELETE data, ensuring privacy.

### Step 5: Create Analysis Views (Optional but Recommended)

Run these SQL statements to create useful views for analyzing plugin performance:

```sql
-- Plugin performance ranking (slowest to fastest)
CREATE OR REPLACE VIEW plugin_performance_ranking AS
SELECT 
    plugin_tested AS plugin,
    COUNT(*) AS scan_count,
    AVG(plugin_speed_delta) AS avg_delta,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY plugin_speed_delta) AS median_delta,
    MIN(plugin_speed_delta) AS min_delta,
    MAX(plugin_speed_delta) AS max_delta,
    SUM(CASE WHEN plugin_speed_delta > 0.3 THEN 1 ELSE 0 END) AS slowdown_count,
    SUM(CASE WHEN error_category = 'break_site' THEN 1 ELSE 0 END) AS break_count,
    SUM(CASE WHEN error_category = 'timeout' THEN 1 ELSE 0 END) AS timeout_count,
    SUM(CASE WHEN error_category = 'output_change' THEN 1 ELSE 0 END) AS output_change_count,
    AVG(baseline_site_load_speed) AS avg_baseline_speed
FROM telemetry
WHERE plugin_speed_delta IS NOT NULL
GROUP BY plugin_tested
ORDER BY avg_delta DESC;

-- Error analysis by plugin
CREATE OR REPLACE VIEW plugin_error_analysis AS
SELECT 
    plugin_tested AS plugin,
    COUNT(*) AS total_scans,
    SUM(CASE WHEN error_category IS NOT NULL AND error_category != 'none' THEN 1 ELSE 0 END) AS error_count,
    SUM(CASE WHEN error_category = 'break_site' THEN 1 ELSE 0 END) AS break_count,
    SUM(CASE WHEN error_category = 'timeout' THEN 1 ELSE 0 END) AS timeout_count,
    SUM(CASE WHEN error_category = 'output_change' THEN 1 ELSE 0 END) AS output_change_count,
    ROUND(
        SUM(CASE WHEN error_category IS NOT NULL AND error_category != 'none' THEN 1 ELSE 0 END)::numeric / 
        COUNT(*)::numeric * 100, 2
    ) AS error_rate_percent
FROM telemetry
GROUP BY plugin_tested;

-- Plugin co-occurrence (which plugins are installed together)
CREATE OR REPLACE VIEW plugin_cooccurrence AS
SELECT 
    p1 AS plugin_a,
    p2 AS plugin_b,
    COUNT(*) AS pair_count
FROM telemetry,
    LATERAL unnest(plugins) WITH ORDINALITY AS t1(p1, ord1),
    LATERAL unnest(plugins) WITH ORDINALITY AS t2(p2, ord2)
WHERE p1 < p2
GROUP BY p1, p2
ORDER BY pair_count DESC;

-- Performance by PHP version
CREATE OR REPLACE VIEW plugin_delta_by_php AS
SELECT 
    plugin_tested AS plugin,
    env->>'php_version' AS php_version,
    COUNT(*) AS scans,
    AVG(plugin_speed_delta) AS avg_delta,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY plugin_speed_delta) AS median_delta
FROM telemetry
WHERE env IS NOT NULL AND plugin_speed_delta IS NOT NULL
GROUP BY plugin_tested, php_version
ORDER BY avg_delta DESC;
```

### Step 6: Configure WordPress Plugin

Add the following to your WordPress plugin's `.env` file:

```ini
PIA_SUPABASE_URL=https://xxxxx.supabase.co
PIA_SUPABASE_ANON_KEY=eyJxxx...
PIA_SUPABASE_TABLE=telemetry
```

Replace the values with the credentials from Step 2.

## Verification

### Check Data is Being Received

1. After a WordPress site scans a plugin with telemetry enabled
2. Run this SQL to verify:

```sql
SELECT * FROM telemetry ORDER BY created_at DESC LIMIT 10;
```

### Check Data Quality

Run this SQL to see submission stats:

```sql
SELECT 
    COUNT(*) AS total_scans,
    COUNT(DISTINCT origin) AS unique_sites,
    MIN(created_at) AS first_scan,
    MAX(created_at) AS latest_scan
FROM telemetry;
```

## Troubleshooting

### "Insert policy missing" error

Ensure the RLS policy from Step 4 is created:

```sql
SELECT policyname, cmd FROM pg_policies WHERE tablename = 'telemetry';
```

### No data appearing

1. Check WordPress plugin telemetry is enabled in admin
2. Check `.env` file has correct Supabase URL and key
3. Check WordPress cron is running: `wp cron event list`
4. Check PHP error logs for failed HTTP requests

### Rate limiting

Supabase Free tier has limits. If you expect high volume:
- Monitor usage in Supabase dashboard
- Consider upgrading to Pro plan
- Implement retry logic with exponential backoff

## Future Enhancements

When scaling up, consider:

1. **Data retention policy**: Add partition by date for managing large datasets
2. **Real-time subscriptions**: Use Supabase Realtime for live dashboards
3. **Edge Functions**: For data transformation before storage
4. **Analytics**: Connect to Metabase or similar for dashboards
